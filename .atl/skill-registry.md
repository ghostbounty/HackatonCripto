| `branch-pr` | `skills/branch-pr` | Preparar la entrega de un change en una rama y un PR coherentes. | `issue-creation` | `skills/issue-creation` | Convertir un change o una parte de sus tasks en issues trazables para seguimiento. | `sdd-apply` | `skills/sdd-apply` | Aplicar cambios en documentacion, codigo o estructura siguiendo el change activo. | `sdd-archive` | `skills/sdd-archive` | Cerrar o archivar un change una vez verificado y fusionado. | `sdd-design` | `skills/sdd-design` | Redactar o ajustar el diseno tecnico de un change en changes/<change-name>/design.md. Usar cuando la spec requiera decisiones de arquitectura, modelo de datos, integracion, persistencia o tradeoffs antes de ejecutar tareas. | - `spec.md` - `docs/architecture.md` - Restricciones tecnicas del repo | - `changes/<change-name>/design.md` | - Los componentes afectados y el flujo tecnico quedan definidos. - El diseno contempla como se sembraran datos demo o se verificaran endpoints y formularios. - Los tradeoffs y supuestos tecnicos estan documentados. - `sdd-tasks` puede particionar el trabajo en slices verificables. | - La spec no es suficientemente clara. - El diseno requerido compromete artifacts preservados o no puede probarse de forma segura. | - `sdd-tasks` |
| `sdd-explore` | `skills/sdd-explore` | Explorar el contexto funcional y tecnico existente antes de proponer cambios. Usar cuando sea necesario leer docs/product, changes vigentes y documentacion marco para comprender alcance, restricciones e impacto. | - `docs/product/` - `changes/<change-name>/` - `docs/architecture.md` - `README.md` - Estado actual del repo | - Resumen de contexto - Riesgos y dependencias - Mapa de impacto para el slice | - Se leyeron los artifacts funcionales obligatorios del dominio. - Se identificaron cambios relacionados y restricciones operativas. - Se documentaron riesgos, dependencias y supuestos relevantes. - El agente puede proponer sin preguntas evitables. | - No se puede localizar el change o los artifacts de dominio necesarios. - Existe ambiguedad critica que el repo no permite resolver. | - `sdd-propose` |
| `sdd-init` | `skills/sdd-init` | Inicializar o alinear la estructura base de un repositorio orientado a specification-driven development. Usar cuando se necesite crear o reparar la capa estructural del repo, verificar la presencia de docs, changes, skills, scripts, .atl o templates de colaboracion. | - Estado actual de la raiz del repo - Contenido existente en `docs/product/` y `changes/` - Reglas operativas definidas en `AGENTS.md` | - `AGENTS.md` - Estructura base del framework - Scripts y docs operativas minimas alineadas | - Las rutas raiz requeridas existen. - `docs/product/` y `changes/` se preservan intactos. - Hay instrucciones claras para instalar, regenerar y validar el framework. - El repo puede pasar a `sdd-explore`. | - Faltan artifacts funcionales preservados. - El alineamiento requerido implicaria romper `docs/product/` o `changes/`. | - `sdd-explore` |
| `sdd-propose` | `skills/sdd-propose` | Redactar o ajustar la propuesta de un change en changes/<change-name>/proposal.md. Usar cuando se necesite definir intencion, problema, alcance, riesgos, rollback y criterios iniciales del cambio. | - Hallazgos de `docs/product/` - Contexto de `changes/` relacionados - Restricciones del repo y del PR | - `changes/<change-name>/proposal.md` | - El problema y el valor del slice quedan claros. - El alcance y fuera de alcance estan definidos. - Se describen riesgos, rollback y criterio inicial de exito. - El slice queda listo para especificarse sin ambiguedad alta. | - No existe claridad minima sobre el problema a resolver. - El slice propuesto contradice `docs/product/` o un change preservado. | - `sdd-spec` |
| `sdd-spec` | `skills/sdd-spec` | Redactar o ajustar la especificacion delta de un change en changes/<change-name>/spec.md. Usar cuando sea necesario formalizar requisitos ADDED, MODIFIED, REMOVED y escenarios de aceptacion sobre la base de docs/product. | - `proposal.md` - Reglas de `docs/product/` - Changes relacionados en `changes/` | - `changes/<change-name>/spec.md` | - Los cambios funcionales estan expresados como deltas claros. - Hay escenarios de aceptacion verificables. - El criterio de prueba del slice queda explicitado. - La spec es consistente con el dominio y el loop autonomo. | - Falta propuesta valida o existe contradiccion con `docs/product/`. - No pueden definirse escenarios verificables. | - `sdd-design` |
| `sdd-tasks` | `skills/sdd-tasks` | Desglosar un change en tareas accionables dentro de changes/<change-name>/tasks.md. Usar cuando se requiera convertir proposal, spec y design en un plan ejecutable y verificable por fases. | - `proposal.md` - `spec.md` - `design.md` - Convencion compartida de tareas | - `changes/<change-name>/tasks.md` - `changes/<change-name>/progress.md` preparado para el primer slice | - El trabajo esta partido en slices pequenos. - Cada slice tiene criterio de aceptacion. - Cada slice declara testing, seeds y verify asociados. - Cada slice puede reportarse de forma clara en `progress.md`. - El orden de ejecucion respeta el loop autonomo. | - Proposal, spec o design no permiten una particion segura. - Las tareas mezclan multiples slices sin gate intermedio. | - `sdd-apply` |
| `sdd-test` | `skills/sdd-test` | Preparar entorno, seeds demo y pruebas Playwright CLI para validar formularios, endpoints y journeys del slice activo. Usar cuando un slice ya fue aplicado y deba demostrar evidencia de pass o fail antes de `sdd-verify`. | - Slice activo ya implementado o alineado - `docs/product/` - `docs/testing-strategy.md` - `docs/seed-fixtures.md` - Scripts de instalacion, setup y validacion - Entorno local necesario para Playwright CLI y datos demo | - Evidencia breve de pass o fail del slice - Estado de seeds demo usadas - Referencia a journeys, formularios y endpoints validados - Hand-off claro para `sdd-verify` - `progress.md` actualizado con resultado `pass`, `retry-needed` o `stop` | - El entorno minimo de testing fue comprobado. - Las seeds demo del marketplace relevantes al slice existen, se prepararon o se verificaron. - Se validaron formularios y endpoints del slice con Playwright CLI o con evidencia equivalente prevista por el repo. - Se registraron claramente resultados, fallas y supuestos. - Existe una conclusion binaria: `pass`, `retry-needed` o `stop`. - `progress.md` deja la ultima evidencia de testing y el siguiente paso exacto. | - No hay forma segura de preparar el entorno minimo. - Faltan seeds criticas para reproducir el flujo. - Los endpoints o formularios del slice no son accesibles o no tienen evidencia suficiente. - Playwright CLI no puede ejecutarse y no existe alternativa documentada para el gate. | - `sdd-verify` si el resultado es `pass` - `sdd-apply` si el resultado es `retry-needed` - `stop` si el resultado es `stop` |
| `sdd-verify` | `skills/sdd-verify` | Verificar que un change y la estructura del repositorio cumplan criterios explicitos. Usar cuando se deban ejecutar checks, completar verify-checklist, validar skills, scripts, testing y compatibilidad con docs/product y changes existentes. | - Cambio aplicado - Evidencia de `sdd-test` - `scripts/validate-structure.sh` - `progress.md` - `verify-checklist.md` o `verify.md` | - Evidencia de validacion - `verify-checklist.md` actualizado si aplica - Resultado del gate: `pass`, `retry-needed` o `stop` - `progress.md` confirmado y actualizado antes del gate | - Las validaciones automaticas y estructurales fueron ejecutadas. - Existe evidencia suficiente de testing del slice. - El repo, el change y el flujo operativo siguen alineados. - Hay una decision clara sobre si se puede avanzar al siguiente slice. - `progress.md` refleja el estado real mas reciente del change. | - Faltan validaciones estructurales o evidencia de testing. - Falta `progress.md` o su estado no coincide con el slice real. - El repo queda inconsistente con `AGENTS.md`, `docs/` o el registry. - Persisten fallas despues de 2 reintentos dirigidos. | - `branch-pr` o `sdd-archive` si el resultado es `pass` y el trabajo termina - `sdd-apply` si el resultado es `retry-needed` - `stop` si el resultado es `stop` |
